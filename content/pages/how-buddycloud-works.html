How Buddycloud Works

Three components that work together: Open SDK, open protocol and open-source server.

You add a small SDK to your mobile or web app; your users communicate.

Web applications then communicate via a websocket with the Buddycloud Server. Mobile applications use the REST API (or XMPP).

Publish-Subscribe

You can think of the Buddycloud server as a very sociable publish-subscribe server. Users messages are fanned out to their followers. Both on the local Buddycloud site and remote buddycloud sites (remember Buddycloud federates?).

Federation

Like Facebook has a wall, Twitter has a Feed, Buddycloud has channels. User's follow channels that can be another user (for example: juliet@capulet.lit follows romeo@montague.lit) or topics (juliet@caputlet.lit follows fan-club@verona-fc.lit).

Your users can follow channels and chat with users on local and remote Buddycloud sites.

Roles

Enable your users' conversation and disable spam. Your application can create users with different roles. Each channel can have an owner, moderator, follower+post, follower, banned. The specifics of each role are defined in the Buddycloud-Protocol.

Channel Nodes

For maximum extensibility each Buddycloud channel is broken down furher into nodes. To keep your application data consistent, each post to the Buddycloud server is validated by a content type plugin. For example the user@example.com/posts node MUST be activity-stream formatted. You can define your own content type - for example a x-application-my-coin-app could validate that each new entry MUST be a valid Bitcoin address.

Channel Metadata

The channel juliet@capulet.lit can have a title Juliet's channel, a description Musings on life in Verona and other metadata and privacy information. The full specification is contained in the Buddycloud protocol document.
